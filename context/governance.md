# Governance Reference — Seven Invariants

Source: Gartner, *How to Govern Anthropic's Claude Code at Scale* (G00850426, March 2026).

This document is the **authoritative reference** for the governance model applied to every project generated by this scaffolder. Claude reads it during generation to know what "done" means.

## Architecture: two layers, three checkpoints

```
┌─────────────────────────┐     ┌──────────────────────────┐
│      Context layer      │     │     Enforcement layer    │
│   (CLAUDE.md, prompts,  │ →   │  (binary pass/fail on    │
│    governance.md)       │     │   the seven invariants)  │
└─────────────────────────┘     └──────────────────────────┘

         Checkpoint 1            Checkpoint 2        Checkpoint 3
        ┌─────────────┐         ┌─────────────┐    ┌──────────────┐
        │ Generation  │   →     │ Pre-commit/ │ →  │    CI/CD     │
        │ Claude hooks│         │  pre-push   │    │   pipeline   │
        │             │         │  Git hooks  │    │    gates     │
        └─────────────┘         └─────────────┘    └──────────────┘
```

All three checkpoints run **the same tools with the same configurations**. Skipping any one creates an exploitable gap.

## The seven invariants

| # | Invariant   | Definition (binary)                                                                       | Tools (this stack)                                                       | Checkpoints       |
|---|-------------|-------------------------------------------------------------------------------------------|--------------------------------------------------------------------------|-------------------|
| 1 | Functional  | Implements the spec; passes static analysis and contract tests.                           | Roslyn analyzers, ESLint, ruff, Semgrep, SonarCloud                      | All three         |
| 2 | Tested      | Unit + integration + E2E tests exist and pass; ≥75% coverage (target 80%+).               | xUnit + coverlet + reportgenerator, Vitest, Playwright                   | Pre-commit, CI/CD |
| 3 | Secure      | No secrets, no known CVEs, no SAST findings, no insecure data flows.                      | gitleaks, dependabot/renovate, CodeQL/Semgrep, (opt) Claude Code Security| All three         |
| 4 | Scalable    | No N+1, no unbounded queries, async I/O on hot paths, idempotent handlers.                | SonarCloud rules, EF Core diagnostics, k6 smoke tests                    | CI/CD             |
| 5 | Performant  | Meets P95 latency budget defined in NFRs; no regressions vs baseline.                     | BenchmarkDotNet, Lighthouse CI, k6                                       | CI/CD             |
| 6 | Observable  | Structured logs, metrics, traces emitted via OpenTelemetry from day one.                  | OpenTelemetry SDK, .NET Aspire dashboards, Serilog                       | CI/CD             |
| 7 | Auditable   | Conventional commits, SBOM, signed artifacts, prompt-lineage ledger of every generation.  | cosign, syft (SBOM), `.claude/provenance/*.jsonl`, conventional-commits  | CI/CD             |

**Rule:** if an artifact fails any invariant, **reject and regenerate**. Never hand-patch generated code — that breaks provenance.

## Pre-commit/push checkpoint

Every generated project ships with a `pre-commit` config (or husky+lint-staged for JS-only) that re-runs:
- formatters (dotnet format, prettier, ruff format, terraform fmt)
- linters (analyzers, eslint, ruff)
- secret scan (gitleaks)
- unit tests on changed files
- conventional-commit message check

## CI/CD checkpoint

Every generated project ships with a `governance` workflow that runs in a clean environment:
1. Build
2. Lint + format check
3. Unit + integration tests + coverage gate (≥75%)
4. SAST (CodeQL or Semgrep)
5. SCA / dependency scan
6. Secret scan (gitleaks)
7. SBOM generation (syft)
8. License compliance check
9. (optional) Reasoning-based scan via Claude Code Security
10. Upload provenance ledger as build artifact

A failure on any invariant blocks the merge.

## Managed settings deployment (non-bypassable)

To make hooks and permissions truly non-bypassable, deploy `managed-settings.json` to the OS-level enterprise path. Developers' local `.claude/settings.json` cannot override these.

| OS      | Managed-policies path                                                       |
|---------|-----------------------------------------------------------------------------|
| macOS   | `/Library/Application Support/ClaudeCode/managed-settings.json`             |
| Linux   | `/etc/claude-code/managed-settings.json`                                    |
| Windows | `C:\ProgramData\ClaudeCode\managed-settings.json`                           |

**Deployment options:**
- macOS: MDM profile (Jamf, Intune, Kandji) pushing the file with root ownership
- Linux: Ansible / Chef / Puppet with `mode: 0644, owner: root`
- Windows: Intune / Group Policy Preference deploying to `ProgramData`

**Verification**: `claude config list --managed` should show the merged effective config. Any developer attempting to disable a hook in their personal settings will see it overridden by the managed layer.

## Visibility & compliance (optional layer)

When the user opts in during the interview:
- **LLM Gateway**: route Claude Code traffic via `ANTHROPIC_BASE_URL` to corporate gateway.
- **OpenTelemetry**: managed config exports usage/decision/event telemetry, segmented by team/cost-center.
- **Prompt lineage**: `.claude/provenance/*.jsonl` per session, archived per release.
- **MCP allowlist**: managed `enabledMcpjsonServers` cannot be widened by developers.
- **Network allowlist**: WebFetch restricted to declared trusted domains.

## IaC / Terraform addendum

For Terraform projects, the seven invariants map to specific tooling:

| Invariant   | Terraform realization                                                                  |
|-------------|----------------------------------------------------------------------------------------|
| Functional  | `terraform fmt -check`, `terraform validate`, `tflint` (with cloud ruleset)            |
| Tested      | `terraform test` (native, 1.6+) per module, plus Terratest for complex scenarios       |
| Secure      | `tfsec` + `checkov` (HIGH+ blocks), `gitleaks`, `block-tf-public-exposure.sh` hook     |
| Scalable    | State segmentation (`one-state-per-stack-per-env`), no monolith state                  |
| Performant  | `-refresh=false` plans, providers cached, no unbounded `for_each`                      |
| Observable  | Diagnostic settings + log analytics baked into modules; alerts for drift               |
| Auditable   | Plan-in-PR artifact, OPA/Conftest policies, Infracost cost diff, drift detection cron, signed module versions, prompt-lineage ledger |

**Non-negotiables for Terraform:**

1. `terraform apply` **never** runs locally — pipeline only, with manual approval on `main` (enforced by `block-terraform-apply.sh`).
2. Every taggable resource carries the mandatory tag set (enforced by `require-tf-tags.sh` + OPA `required_tags.rego`).
3. Cloud auth is **OIDC federated** — no long-lived service-principal secrets in CI.
4. Every `modules/<name>/` directory has `tests/*.tftest.hcl` (enforced by `require-tf-module-tests.sh`).
5. Open CIDRs (`0.0.0.0/0`), `allow_blob_public_access`, `public_network_access_enabled`, disabled encryption, `force_destroy = true` are blocked at generation by `block-tf-public-exposure.sh`. Override only with explicit `# tf-public-exposure: allow` after review.
6. Every PR posts: plan summary, OPA verdict, Infracost diff. Merge blocked if any fail.
7. Drift detection runs on schedule (Q15m) and reports drift as a P1 issue.

## Cautions (Gartner, verbatim intent)

1. **Manual patching breaks provenance.** Reject + regenerate, never hand-tweak.
2. **Skipping any checkpoint creates exploitable gaps.** All three layers run the same tools.
3. **Pattern-matching alone misses business-logic vulnerabilities.** Use reasoning-based scanning where available.
